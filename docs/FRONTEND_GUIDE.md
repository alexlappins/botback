# Гайд по фронту (дашборд и API)

Как залогиниться, работать с шаблонами и отправлять сообщения от имени бота через дашборд и API.

---

## Изоляция клиентов (важно)

Дашборд рассчитан на **клиентов**: каждый пользователь, который зашёл через Discord, видит и может управлять **только своими серверами**.

- Список серверов берётся из Discord по **токену этого пользователя** — возвращаются только те гильдии, где он состоит и у него есть права. Чужие сервера в этот список не попадают.
- Дополнительно показываются только серверы, **где добавлен бот**. В итоге клиент видит лишь сервера, на которых он админ (или с правом управления) и куда приглашён бот.
- Шаблоны и каналы привязаны к серверу (`guildId`). Перед любым действием бэкенд проверяет, что этот сервер входит в список серверов **текущего** пользователя. Запросить шаблоны или отправить сообщение на чужой сервер нельзя — вернётся 403.

Итого: клиент, купивший бота, создаёт и отправляет сообщения только в свои каналы на своих серверах; другие сервера (других клиентов) он не видит и к ним доступа не имеет.

## 1. Вход (логин)

### Через браузер (готовый дашборд)

1. Запусти бота и HTTP-сервер: `npm run start:dev`
2. Открой в браузере: **http://localhost:3000**
3. Нажми **«Войти через Discord»** — откроется страница Discord с запросом прав (профиль, список серверов)
4. Нажми **«Авторизовать»** — тебя вернёт на дашборд, ты будешь залогинен

Сессия хранится в cookie. Чтобы выйти: открой **http://localhost:3000/api/auth/logout** (или ссылка «Выйти» на дашборде).

### Через API (для своего фронта)

- **Начать логин:** редирект пользователя на  
  `GET http://localhost:3000/api/auth/discord`
- **После входа** Discord редиректит на твой `DISCORD_CALLBACK_URL`; бэкенд сам создаёт сессию и редиректит на `/`.
- **Проверить, кто залогинен:**  
  `GET http://localhost:3000/api/auth/me`  
  с **credentials: 'include'** (cookie).  
  Ответ: объект пользователя или 401.

Все запросы к API ниже тоже нужно отправлять с **credentials: 'include'** (или `withCredentials: true`), чтобы уходила сессионная cookie.

### Фронт на отдельном порту (Vite 5173) и прокси

Если фронт крутится на **http://localhost:5173**, а бэкенд на **http://localhost:3000**:

1. **Прокси в проекте фронта** — в бэкенде (этом репо) прокси нет, его настраивают в **твоём** фронт-проекте. Пример для Vite:

   **`vite.config.ts`** (или `vite.config.js`):

   ```ts
   import { defineConfig } from 'vite';

   export default defineConfig({
     server: {
       proxy: {
         '/api': {
           target: 'http://localhost:3000',
           changeOrigin: true,
         },
       },
     },
   });
   ```

   Тогда запрос с фронта на `http://localhost:5173/api/auth/me` уходит на 5173, Vite проксирует его на `http://localhost:3000/api/auth/me`.

2. **Кука сессии и 401** — кука выставляется бэкендом (порт 3000). Чтобы она отправлялась и при запросах на 5173, в **`.env` бэкенда** должен быть указан фронт:

   ```env
   FRONTEND_URL=http://localhost:5173
   ```

   Тогда бэкенд выставляет куку с `domain=localhost`, и браузер будет отправлять её и на 5173. Перезапусти бэкенд и залогинься заново.

3. **Редирект после входа** — с тем же `FRONTEND_URL` бэкенд после логина редиректит на `http://localhost:5173`.

4. **Запросы с фронта** — всегда с куками:

   ```ts
   fetch('/api/auth/me', { credentials: 'include' })
   ```

   Если используешь относительный путь `/api/...`, запрос идёт на 5173 и прокси переправляет его на 3000 — кука при этом уходит (один домен `localhost`).

5. **Почему в Network не видно `/api/auth/discord`**  
   Кнопка входа — это обычная ссылка `<a href={LOGIN_URL}>`. При клике браузер делает **переход по документу**, а не `fetch()`. В Network такой запрос будет с типом **Doc (document)**, а не Fetch/XHR. Если в фильтре выбран только «Fetch/XHR», его не будет в списке — выбери **All** или **Doc**. После запроса сервер отвечает редиректом на Discord, страница меняется и лог может очиститься — включи **Preserve log** в панели Network, чтобы запросы с текущей страницы (в том числе `/api/auth/discord`) оставались в списке после перехода.

## 2. Список серверов (гильдий)

Для залогиненного пользователя возвращаются **только его серверы**: где он состоит, имеет право управлять и где добавлен бот. Другие сервера (других клиентов) в ответ не входят.

**Запрос:**

```
GET http://localhost:3000/api/guilds
```

**Ответ:** массив объектов вида:

```json
[
  {
    "id": "123456789012345678",
    "name": "Название сервера",
    "icon": "abc123...",
    "permissions": "8",
    "owner": false
  }
]
```

Используй `id` как `guildId` в запросах к каналам, отправке сообщений и шаблонам.

---

## 3. Каналы сервера

Нужны для выбора канала при отправке сообщения или шаблона.

**Запрос:**

```
GET http://localhost:3000/api/guilds/:guildId/channels
```

**Ответ:** массив текстовых каналов:

```json
[
  { "id": "987654321098765432", "name": "общий", "type": 0 },
  { "id": "987654321098765433", "name": "анонсы", "type": 0 }
]
```

Подставляй `id` в `channelId` при отправке.

## 4. Отправить сообщение от имени бота (без шаблона)

Сообщение уходит в выбранный канал как эмбед от бота.

**Запрос:**

```
POST http://localhost:3000/api/guilds/:guildId/send
Content-Type: application/json

{
  "channelId": "id_канала",
  "title": "Заголовок эмбеда",
  "description": "Текст описания (можно с переносами строк)",
  "image": "https://example.com/picture.png"
}
```

- `channelId` — обязательно.
- `title`, `description`, `image` — по желанию.

**Ответ:** `200` и `{ "ok": true }` при успехе.

## 5. Шаблоны сообщений (CRUD)

Шаблон — это сохранённый набор полей (название, заголовок, описание, картинка) для одного сервера. По шаблону потом можно отправить сообщение в любой канал одной кнопкой.

### 5.1 Создать шаблон

**Запрос:**

```
POST http://localhost:3000/api/guilds/:guildId/templates
Content-Type: application/json

{
  "name": "Приветствие",
  "title": "Добро пожаловать!",
  "description": "Опиши правила сервера здесь.",
  "image": "https://example.com/welcome.png"
}
```

- `name` — обязательно, уникальное в рамках сервера (например, «Приветствие», «Анонс ивента»).
- Остальные поля можно не передавать или передать `null`.

**Ответ:** объект шаблона с `id`, `guildId`, `name`, `title`, `description`, `image`, `createdAt`, `updatedAt`.

### 5.2 Список шаблонов сервера

**Запрос:**

```
GET http://localhost:3000/api/guilds/:guildId/templates
```

**Ответ:** массив шаблонов (от новых к старым по `updatedAt`).

### 5.3 Получить один шаблон

**Запрос:**

```
GET http://localhost:3000/api/guilds/:guildId/templates/:templateId
```

**Ответ:** объект шаблона или 404.

### 5.4 Обновить шаблон

**Запрос:**

```
PATCH http://localhost:3000/api/guilds/:guildId/templates/:templateId
Content-Type: application/json

{
  "name": "Новое название",
  "title": "Новый заголовок",
  "description": "Новый текст",
  "image": "https://..."
}
```

Можно передать только те поля, которые меняются.

**Ответ:** обновлённый объект шаблона.

### 5.5 Удалить шаблон

**Запрос:**

```
DELETE http://localhost:3000/api/guilds/:guildId/templates/:templateId
```

**Ответ:** `200` и `{ "ok": true }`.

---

## 6. Отправить сообщение из шаблона от имени бота

Подставляются сохранённые в шаблоне `title`, `description`, `image`; сообщение отправляется в выбранный канал.

**Запрос:**

```
POST http://localhost:3000/api/guilds/:guildId/templates/:templateId/send
Content-Type: application/json

{
  "channelId": "id_канала"
}
```

**Ответ:** `200` и `{ "ok": true }` при успехе.

В Discord в указанном канале появится сообщение от бота с эмбедом по данным шаблона.

## 6.1 Логи (каналы для событий)

Настройка каналов, в которые бот отправляет логи (вход/выход, сообщения, модерация, каналы, бан/кик). Можно задать **в дашборде** (на странице сервера — блок «Каналы для логов»), через **API** (PATCH ниже) или слэш-команду `/logs set` в Discord.

### Получить настройки логов

**Запрос:**

```
GET http://localhost:3000/api/guilds/:guildId/logs
```

**Ответ:** объект с ключами типов логов и ID каналов (если заданы):

```json
{
  "joinLeave": "1234567890123456789",
  "messages": "9876543210987654321",
  "moderation": null,
  "channel": null,
  "banKick": null
}
```

Типы: `joinLeave`, `messages`, `moderation`, `channel`, `banKick`. Отсутствующие ключи или `null` — канал не задан.

### Задать канал для типа логов

**Запрос:**

```
PATCH http://localhost:3000/api/guilds/:guildId/logs
Content-Type: application/json

{
  "type": "joinLeave",
  "channelId": "1234567890123456789"
}
```

Чтобы отключить логи типа, передай `channelId: null`:

```json
{
  "type": "messages",
  "channelId": null
}
```

**Ответ:** обновлённый объект логов (как в GET).

### Лента событий (история логов)

События пишутся в БД при каждом срабатывании (вход/выход, удаление сообщения, бан, таймаут и т.д.). Хранятся не более **3 месяцев**, затем удаляются по крону.

**Запрос:**

```
GET http://localhost:3000/api/guilds/:guildId/logs/events?limit=50&before=uuid
```

- `limit` — сколько записей вернуть (по умолчанию 50, макс. 100).
- `before` — UUID последней записи из предыдущей выборки (курсор для пагинации «назад в прошлое»).

**Ответ:**

```json
{
  "events": [
    {
      "id": "uuid",
      "guildId": "123...",
      "type": "joinLeave",
      "kind": "member_join",
      "payload": { "userId": "...", "userTag": "User#0", "avatarUrl": "https://..." },
      "createdAt": "2025-03-09T20:00:00.000Z"
    }
  ]
}
```

Поля события:
- `type` — категория: `joinLeave`, `messages`, `moderation`, `channel`, `banKick`.
- `kind` — конкретное событие: `member_join`, `member_leave`, `member_kick`, `message_delete`, `message_edit`, `channel_create`, `channel_delete`, `ban_add`, `timeout`, `timeout_remove`.
- `payload` — объект с полями для отображения (зависит от `kind`), например `userTag`, `userId`, `reason`, `contentPreview`, `untilTimestamp` и т.д.

## 6.2 Роли по реакции (привязки эмодзи → роль)

Список привязок «реакция на сообщение → роль», добавление и снятие через API или в дашборде (на странице сервера). В Discord тоже можно: `/reaction-role-emoji`.

### Список ролей сервера (для выбора в формах)

**Запрос:** `GET /api/guilds/:guildId/roles`  
**Ответ:** массив `{ id, name }` (роли, которые видит бот, без @everyone).

### Получить список привязок

**Запрос:**

```
GET http://localhost:3000/api/guilds/:guildId/reaction-roles
```

**Ответ:**

```json
{
  "bindings": [
    {
      "messageId": "111222333444555666",
      "channelId": "777888999000111222",
      "roles": [
        { "emojiKey": "✅", "roleId": "333444555666777888" },
        { "emojiKey": "1234567890123456", "roleId": "999000111222333444" }
      ]
    }
  ]
}
```

- `messageId` — ID сообщения в Discord.
- `channelId` — ID канала (если известен боту; для старых привязок может отсутствовать).
- `roles` — массив пар: `emojiKey` (unicode-символ или ID кастомного эмодзи), `roleId` — ID роли.

Ссылка на сообщение в Discord: `https://discord.com/channels/{guildId}/{channelId}/{messageId}`.

### Снять привязку (эмодзи → роль)

**Запрос:**

```
POST http://localhost:3000/api/guilds/:guildId/reaction-roles/remove
Content-Type: application/json

{
  "messageId": "111222333444555666",
  "emojiKey": "✅"
}
```

Для кастомного эмодзи `emojiKey` — это ID эмодзи (числовая строка).

**Ответ:** `200` и `{ "ok": true }`.

### Добавить привязку (эмодзи → роль)

**Запрос:**

```
POST http://localhost:3000/api/guilds/:guildId/reaction-roles
Content-Type: application/json

{
  "channelId": "id_канала",
  "messageId": "id_сообщения",
  "emoji": "✅",
  "roleId": "id_роли"
}
```

Бот поставит реакцию на указанное сообщение в канале и сохранит привязку. Эмодзи: unicode (например `✅`) или для кастомного — `имя:id`.

**Ответ:** `200` и `{ "ok": true }` при успехе.

## 7. Типичный сценарий на фронте

1. **Логин**  
   Редирект на `/api/auth/discord` или кнопка «Войти через Discord» с этим URL.

2. **Главная**  
   Запрос `GET /api/auth/me`. Если 401 — показать кнопку входа. Если 200 — показать список серверов через `GET /api/guilds`.

3. **Страница сервера**  
   По клику на сервер открыть страницу с `guildId` из списка.  
   - Загрузить каналы: `GET /api/guilds/:guildId/channels`.  
   - Загрузить шаблоны: `GET /api/guilds/:guildId/templates`.  
   - При необходимости: логи — `GET /api/guilds/:guildId/logs`, привязки ролей по реакции — `GET /api/guilds/:guildId/reaction-roles`.

4. **Создание шаблона**  
   Форма: название + заголовок + описание + URL картинки.  
   Отправка: `POST /api/guilds/:guildId/templates` с этими полями.  
   После успеха — обновить список шаблонов (или перезапросить `GET .../templates`).

5. **Отправка из шаблона**  
   У шаблона кнопка «Отправить» → модалка/форма выбора канала (из `GET .../channels`).  
   Отправка: `POST /api/guilds/:guildId/templates/:templateId/send` с `{ "channelId": "..." }`.

6. **Редактирование / удаление**  
   Редактирование: форма с текущими полями шаблона → `PATCH /api/guilds/:guildId/templates/:templateId`.  
   Удаление: подтверждение → `DELETE /api/guilds/:guildId/templates/:templateId`.

---

## 8. Важно для фронта

- Все запросы к API (кроме первого редиректа на Discord) делай с **credentials: 'include'** (fetch) или **withCredentials: true** (axios), чтобы отправлялась cookie сессии.
- Если получил **401** или **403** — покажи пользователю «Войдите» или «Нет доступа к этому серверу».
- `guildId` и `channelId` в Discord — это снежинки (длинные строки из цифр), передавай их как строки.
- Сообщения всегда отправляются **от имени бота** в выбранный канал; пользователь только выбирает канал и (при шаблонах) какой шаблон отправить.

Если что-то из этого нужно вынести в отдельный раздел (например, только шаблоны или только логин) — можно разбить гайд на несколько файлов.

## 9. Краткая справка для фронта (что нужно)

### Окружение и запросы

| Что | Значение |
|-----|----------|
| Бэкенд | `http://localhost:3000` (или твой хост) |
| Фронт (dev) | например `http://localhost:5173` |
| Прокси (Vite) | `proxy: { '/api': { target: 'http://localhost:3000' } }` |
| Все запросы к API | с **credentials: 'include'** (cookie сессии) |
| Базовый путь к API | `/api` (при прокси с 5173 — запросы на свой origin, прокси переправляет на 3000) |

### Авторизация

| Действие | Как |
|----------|-----|
| Вход | Редирект пользователя на `GET /api/auth/discord` (или ссылка на этот URL) |
| Кто залогинен | `GET /api/auth/me` → 200 + `{ id, username, avatar, discriminator }` или 401 |
| Выход | Переход по `GET /api/auth/logout` (редирект на главную/фронт) |

После логина бэкенд редиректит на `FRONTEND_URL` (например `http://localhost:5173`). В ответе на callback приходит **Set-Cookie** с сессией — дальше все запросы с `credentials: 'include'` будут авторизованы.

### API (все методы требуют авторизации)

| Метод | URL | Тело (JSON) | Ответ |
|-------|-----|------------|--------|
| GET | `/api/auth/me` | — | Пользователь или 401 |
| GET | `/api/guilds` | — | Массив `{ id, name, icon, permissions, owner }` |
| GET | `/api/guilds/:guildId/channels` | — | Массив `{ id, name, type }` |
| POST | `/api/guilds/:guildId/send` | `{ channelId, title?, description?, image? }` | `{ ok: true }` |
| GET | `/api/guilds/:guildId/templates` | — | Массив шаблонов |
| POST | `/api/guilds/:guildId/templates` | `{ name, title?, description?, image? }` | Шаблон |
| GET | `/api/guilds/:guildId/templates/:id` | — | Один шаблон |
| PATCH | `/api/guilds/:guildId/templates/:id` | Частичные поля | Шаблон |
| DELETE | `/api/guilds/:guildId/templates/:id` | — | `{ ok: true }` |
| POST | `/api/guilds/:guildId/templates/:id/send` | `{ channelId }` | `{ ok: true }` |
| GET | `/api/guilds/:guildId/logs` | — | Объект типов логов → channelId |
| GET | `/api/guilds/:guildId/logs/events` | query: `limit`, `before` | `{ events: LogEvent[] }` |
| PATCH | `/api/guilds/:guildId/logs` | `{ type, channelId \| null }` | Обновлённые логи |
| GET | `/api/guilds/:guildId/roles` | — | Массив `{ id, name }` (роли сервера) |
| GET | `/api/guilds/:guildId/reaction-roles` | — | `{ bindings: [{ messageId, channelId?, roles: [{ emojiKey, roleId }] }] }` |
| POST | `/api/guilds/:guildId/reaction-roles` | `{ channelId, messageId, emoji, roleId }` | `{ ok: true }` |
| POST | `/api/guilds/:guildId/reaction-roles/remove` | `{ messageId, emojiKey }` | `{ ok: true }` |

Типы логов для PATCH: `joinLeave`, `messages`, `moderation`, `channel`, `banKick`. Привязки «роль по реакции» можно добавлять в дашборде (POST reaction-roles) или в Discord: `/reaction-role`, `/reaction-role-emoji`.

## 10. Что вывести в дашборде (логи и роли по реакции)

API уже отдаёт всё нужное для отображения **настроек** (какие каналы выбраны для логов, какие привязки реакций есть). Сам поток событий (кто зашёл, что удалили и т.д.) в API не отдаётся — он идёт только в Discord-каналы.

### 10.1 Блок «Логи»

**Данные:** `GET /api/guilds/:guildId/logs` → объект вида `{ joinLeave: "channelId", messages: "…", … }`.

**Что показать:**

- Таблицу или карточки по типам логов:
  - **Тип** — подпись из маппинга ниже.
  - **Канал** — если есть `channelId`, показать имя канала (взять из `GET /api/guilds/:guildId/channels` по `id`) или «#канал» по ID; если нет — «Не настроен».
  - **Действие** — кнопка «Изменить» / «Выбрать канал» или «Отключить».

**Маппинг тип → подпись для UI:**

| Ключ в API   | Подпись для дашборда   |
|--------------|------------------------|
| `joinLeave`  | Вход / Выход           |
| `messages`   | Сообщения              |
| `moderation` | Модерация (таймауты)   |
| `channel`    | Каналы                 |
| `banKick`    | Бан / Кик              |

**Изменение:** при выборе канала из списка каналов сервера отправить  
`PATCH /api/guilds/:guildId/logs` с телом `{ type: "joinLeave", channelId: "…" }` (или другой тип). Чтобы отключить — `channelId: null`. После ответа обновить блок (повторный GET или использовать ответ PATCH).

### 10.2 Блок «Роли по реакции»

**Данные:** `GET /api/guilds/:guildId/reaction-roles` → `{ bindings: [ { messageId, channelId?, roles: [ { emojiKey, roleId } ] } ] }`.

**Что показать:**

- Список привязок. Для каждой:
  - **Сообщение** — если есть `channelId`, ссылка в Discord:  
    `https://discord.com/channels/{guildId}/{channelId}/{messageId}` (открыть в новой вкладке). Иначе текст вида «ID сообщения: …».
  - **Эмодзи → роль** — для каждого элемента в `roles`: отобразить `emojiKey` (символ или подпись «Кастомный») и роль (по `roleId` можно показать ID; имени роли в API нет — при желании можно расширить бэкенд).
  - **Удалить** — кнопка «Удалить» для одной пары эмодзи–роль:  
    `POST /api/guilds/:guildId/reaction-roles/remove` с телом `{ messageId, emojiKey }`. После успеха обновить список (повторный GET или убрать строку с фронта).

**Добавление привязки:** в API не вынесено — делается в Discord командой `/reaction-role-emoji`. В дашборде можно показать подсказку: «Новые привязки настраиваются в Discord: `/reaction-role-emoji`».

### 10.3 Лента событий

**Данные:** `GET /api/guilds/:guildId/logs/events?limit=50&before=uuid`.

**Что показать:** список событий по убыванию даты (`createdAt`). Для каждого — иконка/подпись по `kind`, дата, и поля из `payload` (ник, причина, превью текста и т.д.). Кнопка «Ещё» / подгрузка: передать `before` = `id` последнего события в списке.

**Ретеншн:** в БД хранятся только события за последние **3 месяца** (очистка по крону раз в сутки).

### 10.4 Чего в дашборде нет

- **Имена ролей** в ответе reaction-roles — приходят только `roleId`. При необходимости можно добавить в ответ поля с именами ролей (бот может взять их из кеша гильдии).
